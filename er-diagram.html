Is it an RDBMS?
Yes. PostgreSQL is a Relational Database Management System — relationships are foreign keys enforced by the database at write time, not application code. You cannot insert a ticket for a tenant that doesn't exist, or delete a tenant while its tickets still reference it, without an explicit cascade. That enforcement is the entire point of choosing an RDBMS here.

Relationships that matter, by module
core — tenancy & identity
  • tenants → users, contacts, roles, teams, approval_groups
  • roles ↔ users (via user_roles)
  • roles ↔ permissions (via role_permissions)
  • teams ↔ users (via team_members)
  • approval_groups ↔ users (via approval_group_members)
itsm — tickets, problems, changes
  • contacts → tickets (requester)
  • users → tickets (assignee)
  • tickets → ticket_activity, attachments, links
  • tickets ↔ problems (via problem_incidents)
  • tickets → change_requests (source_ticket_id)
  • change_requests → change_approvals ← approval_groups
  • ticket_categories, kb_categories, service_catalog_categories → themselves (nested, self-referencing)
dev — sprints & backlog
  • projects → backlog_items, epics, sprints, scrum_teams
  • epics, sprints → backlog_items
  • tickets, problems → backlog_items (linked work)
  • backlog_items ↔ change_requests — two-way link, see note below
  • change_requests → deployments
sam — assets
  • assets → asset_assignments ← users
  • assets → license_renewals
  • users → assets (owner_user_id)
docs — files & wiki
  • folders → documents → document_versions
  • folders → folders (nested, self-referencing)
  • wiki_spaces → wiki_pages → history, comments
  • wiki_pages → wiki_pages (nested, self-referencing)
platform — plans & billing
  • plans → subscriptions ← tenants
  • tenants → tenant_entitlements
  • platform.services stands alone — cross-tenant health, no tenant_id
One genuinely unusual pair: dev.backlog_items.change_request_id and itsm.change_requests.backlog_item_id point at each other — a backlog card can name "its" change request, and a change request separately records which card it promotes. That mirrors the app's own UI, where you can link a card to a CR from either side.
Every table above (and 30+ more not listed — attachments, audit rows, settings) also carries its own tenant_id → core.tenants.id foreign key — the single most repeated relationship in the schema, and what makes the row-level multi-tenancy design in Technical design enforceable by the database itself, not just application code.
