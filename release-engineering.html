Release engineering & disaster recovery
Branching strategy, environment creation (first-time and ongoing), the code-promotion pipeline, testing gates, production updates, and DR — how a change gets from a developer's laptop to every customer safely, and what happens if a whole region goes down.
Application architecture → Cloud hosting →
In one line
Trunk-based development with short-lived branches, four environments (Dev, QA, Staging, Production) provisioned identically from the same Infrastructure-as-Code modules, one build artifact promoted unchanged through all of them behind testing gates, and a paired-region DR setup with async replication so a full region loss is a promotable standby, not a rebuild from backup.
Branching strategy
Trunk-based development — main is always deployable, branches are short-lived (days, not weeks), and incomplete work ships behind feature flags instead of living on a branch. Long-lived branches are the thing this avoids: the longer a branch lives, the more merge pain it stores up.
  • Branch naming — feature/<ticket-id>-short-desc, fix/<ticket-id>-short-desc, cut directly from main.
  • Every PR requires — passing CI (build, unit tests, lint, SAST/dependency scan), at least one review, no direct pushes to main.
  • Feature flags decouple "merged" from "released" — a half-finished feature merges to main continuously behind a flag that's off in production, instead of sitting unmerged on a branch for weeks.
  • Hotfix path — branch directly off the current production tag, PR back into main and cherry-picked forward if main has diverged. Rare in practice, since trunk-based keeps main ≈ production most of the time.
Code promotion pipeline
One build artifact — never rebuilt per environment, only reconfigured (connection strings, feature flags) — is promoted through every stage. What passed testing in Staging is byte-for-byte what ships to Production.
Container Apps revisions provide canary/blue-green natively: a new revision takes a small slice of traffic, ramps up automatically while health probes and error rate stay clean, and the dashed line back from Monitoring is the automatic rollback signal — a bad revision gets pulled without a human needing to notice first.

First-time infra creation
One set of Bicep/Terraform modules — matching the layers in Cloud hosting — parameterized per environment. Running terraform apply (or az deployment group create for Bicep) with an environment-specific parameters file creates that environment's entire resource group from scratch: DNS, Front Door, Container Apps environment, Postgres Flexible Server, Redis, Storage, Key Vault, Service Bus. Secrets are generated straight into Key Vault as part of that same apply — never hand-typed anywhere.
Ongoing infra updates
Same modules, same command. terraform plan / bicep what-if runs in CI on every infra-repo PR so reviewers see the exact diff before anything applies. Apply goes through the same pipeline gates as code — auto-apply to Dev, approval required for Staging/Production. A nightly drift-detection plan (expecting zero changes) catches manual portal edits that bypassed the pipeline.

Production update process
  • Build once — the exact artifact that passed Staging is what deploys to Production, never rebuilt per environment.
  • DB migrations use expand-contract — a backward-compatible migration (new columns/tables) deploys before the code that uses them; old columns are dropped only after the new code path has run clean in production for a verified period. A mid-rollout rollback never leaves the database in a shape the old code can't read.
  • Canary rollout — the new Container Apps revision takes a small percentage of traffic first, ramping automatically while health probes and error rate stay within budget.
  • Feature flags gate user-facing behavior separately from deployment — "deployed" and "released" are different events, reversible without a redeploy.
  • Automatic rollback — if health probes or error rate cross a threshold during ramp-up, the canary revision is pulled and traffic reverts to the last known-good revision without waiting for a human to notice.
Disaster recovery strategy
Primary region active, a paired region on standby — async replication keeps it current, and Front Door fails over on health-probe failure without a manual DNS change.

  • Postgres Flexible Server — geo-redundant backups plus a replica in the DR region, promotable to primary on failover.
  • Blob Storage — geo-redundant storage (GRS/RA-GRS); data is replicated automatically, no separate DR job to run.
  • Redis — not on the DR-critical path. It's a cache; the DR region starts with an empty one and rebuilds from Postgres, which is acceptable since sessions can simply be re-established.
  • Front Door — automatic health-probe-based failover to the DR region; no manual DNS cutover required.
  • DR drills — a quarterly failover game-day exercises the actual runbook and verifies real RTO/RPO against the targets above, not just the theoretical design.
