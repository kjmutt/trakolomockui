Cloud hosting
Where each layer of the application architecture actually runs — a managed-Azure reference stack, an AWS comparison for the layers that are cloud-agnostic, and the standalone on-premise install for the perpetual-license edition.

In one line
Same six layers as the application architecture, mapped onto managed Azure services — Front Door at the edge, Container Apps for the fleet and the five job-type workers, Flexible Server for the hybrid data tier, Service Bus for background dispatch. Nothing about the app changes; only which cloud runs it.

Layer walkthrough
①Edge & routing
Front Door terminates a wildcard managed TLS cert and load-balances to the app fleet — CDN, WAF, and global routing in one resource, where AWS splits the same job across ALB/CloudFront.
Azure DNS · *.trakolo.com Front Door — managed TLS + WAF portal.trakolo.com — sign-in
▼
②App fleet
Static Web Apps serves the React SPA; Container Apps runs the .NET Core API as a modular monolith, scaling on HTTP load via KEDA. Shared compute pool for every tenant by default — compute_pool pins a rare contractually-demanding tenant to a dedicated Container Apps environment without a second codebase.
Static Web Apps — apps/web (SPA) Container Apps — apps/api (itsm · sam · dev · docs) compute_pool: shared (default) / dedicated (opt-in)
▼
③Control plane
Same Redis protocol as the AWS reference — the tenant-registry cache client code doesn't change between clouds.
Tenant registry (slug → db host, bucket, plan, compute_pool) Azure Cache for Redis
▼
④Data tier
Azure Database for PostgreSQL Flexible Server, zone-redundant HA, fronted by a PgBouncer sidecar — shared instance with row-level tenant_id scoping (see db/schema.sql) for Free/Team/Business, a dedicated Flexible Server per Enterprise tenant, and a read replica isolating reporting from ticket-write traffic.
PgBouncer sidecar Flexible Server — shared, zone-redundant HA Flexible Server — dedicated per Enterprise tenant Read replica — reporting Blob Storage — shared container + prefix, dedicated for Enterprise
▼
⑤Background & specialized services
Five independent Container Apps jobs, each with its own Service Bus queue — not one shared worker pulling off one queue. A crash or bad deploy in worker-webhooks never stops SLA escalation from running for every other customer; idempotency keys plus Service Bus session locks make a redelivered message a no-op instead of a duplicate.
worker-sla — SLA escalation worker-rules — business rules engine worker-webhooks — delivery + retries worker-reports — export scheduler worker-ai-agent — separate service, different resource profile
▼
⑥Monitoring & secrets
Azure Monitor and Application Insights feed the same health signals as Admin → System status, per worker, routed to PagerDuty. Key Vault holds DB passwords, API keys, and the Trakolo private key used to sign .trklic license files — never a plain .env in production.
Azure Monitor + Application Insights Alerts → PagerDuty, per worker Key Vault — secrets
Layer-by-layer: AWS vs. Azure
LayerAWS referenceAzure equivalentWhy it's a drop-in swap
EdgeRoute 53, ACM, ALB/CloudFrontAzure DNS, Front Door (managed TLS + WAF)Both resolve a subdomain, terminate TLS, and route to the app fleet — no app code touches this layer.
App fleetECS Fargate / EKSContainer Apps (or AKS) + Static Web AppsContainer Apps auto-scales on HTTP/CPU load the same way Fargate does — the .NET Core API image doesn't change.
Control planeElastiCacheAzure Cache for RedisSame Redis protocol — the tenant-registry cache client code is identical either way.
Data tierRDS Postgres Multi-AZ + read replica, S3Flexible Server (zone-redundant HA) + read replica, Blob StorageBoth are managed Postgres with automated failover and backups — db/schema.sql runs unmodified on either.
Background services5 separate ECS services (sla, rules, webhooks, reports, ai-agent), each SQS-backed or Postgres-backed5 separate Container Apps jobs, each on its own Service Bus queueWorkers only need a queue to pull from — the job-processing code, including the idempotency-key check, is cloud-agnostic per worker.
MonitoringCloudWatch / Datadog → PagerDutyAzure Monitor + Application Insights → PagerDutyBoth ingest the same per-worker health signals already shown in Admin → System status.
SecretsSecrets ManagerAzure Key VaultDB passwords, API keys, and the license-signing key are read at boot from a managed vault either way — never committed, never in a plain .env in production.
The only layer where the two clouds genuinely diverge in behavior is Front Door vs. ALB/CloudFront — Front Door bundles CDN, WAF, and global load balancing in one resource, where AWS splits that across two or three. Everything else is a like-for-like managed-service swap; the same shape maps onto Cloud SQL/Memorystore/GCS on GCP too.

License activation
A workspace's .trklic file — a JSON payload signed with Trakolo's private key — is issued whenever seats, modules, or the expiry date change, and verified locally against the public key bundled with the install. Air-gapped networks activate offline: License → Offline activation generates a request code, exchanged for a response code on the customer portal from any machine with outbound access, then pasted back in. See site/docs.html for the full walkthrough.
