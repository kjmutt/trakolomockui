System requirements
Service versions and hardware sizing for the standalone/on-premise install (install/docker-compose.yml), plus capacity planning for the multi-tenant cloud SaaS tier by concurrent-user count — current as of July 2026, re-check the source links before every release.

In one line
Postgres 16 (pinned) is fine through 2028 but Postgres 18 is the current stable if starting fresh; Redis should move from the 7.x pin to 8.x; nginx's floating alpine tag needs no action. Hardware scales with concurrent users, not company size — Postgres is the resource-hungry piece.

Hardware sizing — single-host on-prem install
This is the perpetual-license edition — one host, one docker compose up, no Kubernetes — so sizing is per customer, not the 10,000-concurrent multi-tenant SaaS scale in Cloud hosting. Postgres is the RAM-hungry piece; disk should be SSD/NVMe specifically because ticket search and Dev-module diff/blob operations are I/O-heavy.

Architecture inflection—dedicated Enterprise DBs start appearingroughly the "10K concurrent" baseline the current design targetsdedicated-DB-per-Enterprise becomes operational necessity; largest shared tables (tickets, audit logs) need partitioning by tenant_id or timeshared tier must shard — split into 4–8 Postgres clusters by tenant_id range, each sized like a 10K–50K-tier DB on its own
Every row scales linearly except the shared Postgres primary at 50K+ — no single box, however large, clears the write-throughput and connection fan-in ceiling past that point. That's the one place this is an architecture change (sharding the shared tier), not just a bigger VM. These are informed capacity-planning estimates from standard Postgres/Redis/app-server sizing heuristics, not benchmarked numbers for this product — actual load depends heavily on attachment/file volume, tenant activity distribution, and per-module query patterns.
