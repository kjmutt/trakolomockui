Technical design
Reference architecture for building Trakolo as a real product — multi-tenant, subdomain-routed, sized for 10,000 concurrent users. Not part of the product itself; internal engineering documentation.
View db/schema.sql →
In one line
One modular-monolith app fleet behind wildcard subdomains, a hybrid data tier — shared clusters for most tenants, dedicated database & storage for Enterprise — and background services that scale independently of the request path.
Request & data flow
①Edge & routing
One wildcard DNS record and TLS cert cover every tenant subdomain. portal.trakolo.com is the tenant-agnostic sign-in chooser — the same job login.html does in this mockup.
Wildcard DNS · *.trakolo.com Load balancer + wildcard TLS portal.trakolo.com — sign-in
▼
②App fleet — stateless, horizontally scaled
No per-tenant pinning — any pod serves any tenant's request once it's resolved. The API stays a modular monolith: one deployable, module boundaries as internal packages.
apps/web (SPA) apps/api — itsm apps/api — sam apps/api — dev apps/api — docs

③Control plane
Resolves subdomain → tenant_id → DB host + storage bucket on every request, cached so tenant resolution never costs a database round-trip.
Tenant registry (slug → db host, bucket, plan) Redis — tenant cache + sessions
▼
④Data tier
Shared, pooled cluster with row-level tenant_id scoping (see db/schema.sql) for Free/Team/Business; a dedicated database + bucket per Enterprise tenant. A read replica isolates reporting from ticket-write traffic.
PgBouncer / RDS Proxy Shared Postgres — Free · Team · Business Dedicated Postgres — per Enterprise tenant Read replica — reporting Object storage — shared bucket + prefix, dedicated for Enterprise
▼
⑤Background services — apps/worker
Deploy and scale independently of the request/response app tier, off a shared job queue. These are the exact jobs already listed in Admin → System status.
Job queue (Postgres-backed) SLA escalation Business rules engine AI agent worker Webhook delivery Report export scheduler
▼
⑥Monitoring
The same health registry behind Admin → System status feeds real dashboards and alerts in production.
platform.services — health registry Dashboards + alerts

Two things this design depends on
Route everything through the registry
No service hardcodes which database or bucket a tenant lives in — every lookup goes through the control-plane registry. That's what makes "shared for most, dedicated for Enterprise" a config difference instead of two codebases.
Shared infrastructure shares fate
Tenants on the shared cluster share connection pool, disk I/O, and query load. Per-tenant rate limits and the reporting read replica exist specifically to stop one noisy tenant from becoming everyone else's incident.
