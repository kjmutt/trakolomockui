Application architecture
Reference architecture for building Trakolo as a real product — multi-tenant, subdomain-routed, sized for 10,000 concurrent users. Not part of the product itself; internal engineering documentation.
Entity relationships → View db/schema.sql →
In one line
One modular-monolith app fleet behind wildcard subdomains, a hybrid data tier — shared cluster for most tenants, dedicated database & storage for Enterprise — and background services split by job type so one failure never takes down every job for every customer.

Layer walkthrough
①Edge & routing
One wildcard DNS record and TLS cert cover every tenant subdomain. portal.trakolo.com is the tenant-agnostic sign-in chooser — the same job login.html does in this mockup.
Wildcard DNS · *.trakolo.com Load balancer + wildcard TLS portal.trakolo.com — sign-in
▼
②App fleet — stateless, horizontally scaled
Shared compute pool for every tenant by default, including Enterprise — no per-tenant pinning, any instance serves any tenant's request once resolved. A compute_pool field on the tenant registry is the escape hatch: it can pin one contractually-demanding tenant to a dedicated pool without a second codebase.
apps/web (SPA) apps/api — itsm apps/api — sam apps/api — dev apps/api — docs compute_pool: shared (default) / dedicated (opt-in)
▼
③Control plane
Resolves subdomain → tenant_id → DB host + storage bucket + compute_pool on every request, cached so tenant resolution never costs a database round-trip.
Tenant registry (slug → db host, bucket, plan, compute_pool) Redis — tenant cache + sessions
▼
④Data tier
Shared, pooled cluster with row-level tenant_id scoping (see db/schema.sql) for Free/Team/Business; a dedicated database + bucket per Enterprise tenant. A read replica isolates reporting from ticket-write traffic.
PgBouncer / RDS Proxy Shared Postgres — Free · Team · Business Dedicated Postgres — per Enterprise tenant Read replica — reporting Object storage — shared bucket + prefix, dedicated for Enterprise
▼
⑤Background & specialized services
Split by job type into independent deployables — not one shared worker process. If webhook delivery crashes, SLA escalation keeps running for every customer; it doesn't take everyone down at once. Each job carries an idempotency key and dequeues with FOR UPDATE SKIP LOCKED-style locking so a redelivered message is a no-op, not a duplicate.
worker-sla — SLA escalation worker-rules — business rules engine worker-webhooks — delivery + retries worker-reports — export scheduler worker-ai-agent — separate service, different resource profile
▼
⑥Monitoring
The same health registry behind Admin → System status feeds real dashboards and alerts in production — now per worker, so a failure is attributed to one job type, not "background services" generically.
platform.services — health registry Dashboards + alerts, per worker

Things this design depends on
Route everything through the registry
No service hardcodes which database, bucket, or compute pool a tenant lives in — every lookup goes through the control-plane registry. That's what makes "shared for most, dedicated for one" a config difference instead of two codebases.
Shared infrastructure shares fate
Tenants on the shared cluster share connection pool, disk I/O, and query load. Per-tenant rate limits and the reporting read replica exist specifically to stop one noisy tenant from becoming everyone else's incident.
One worker, one failure domain
Splitting background jobs by type means a crash, a bad deploy, or a stuck queue in one worker is contained to that job — the lesson from the previous design, where one shared worker process going down stopped every background job for every customer at once.
