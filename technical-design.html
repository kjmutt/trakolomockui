Technical design
Reference architecture for building Trakolo as a real product — multi-tenant, subdomain-routed, sized for 10,000 concurrent users. Not part of the product itself; internal engineering documentation.
Entity relationships → View db/schema.sql →
In one line
One modular-monolith app fleet behind wildcard subdomains, a hybrid data tier — shared clusters for most tenants, dedicated database & storage for Enterprise — and background services that scale independently of the request path.
Request & data flow
①Edge & routing
One wildcard DNS record and TLS cert cover every tenant subdomain. portal.trakolo.com is the tenant-agnostic sign-in chooser — the same job login.html does in this mockup.

②App fleet — stateless, horizontally scaled
No per-tenant pinning — any pod serves any tenant's request once it's resolved. The API stays a modular monolith: one deployable, module boundaries as internal packages.

③Control plane
Resolves subdomain → tenant_id → DB host + storage bucket on every request, cached so tenant resolution never costs a database round-trip.

④Data tier
Shared, pooled cluster with row-level tenant_id scoping (see db/schema.sql) for Free/Team/Business; a dedicated database + bucket per Enterprise tenant. A read replica isolates reporting from ticket-write traffic.

Deploy and scale independently of the request/response app tier, off a shared job queue. These are the exact jobs already listed in Admin → System status.

Two things this design depends on
Route everything through the registry
No service hardcodes which database or bucket a tenant lives in — every lookup goes through the control-plane registry. That's what makes "shared for most, dedicated for Enterprise" a config difference instead of two codebases.
Shared infrastructure shares fate
Tenants on the shared cluster share connection pool, disk I/O, and query load. Per-tenant rate limits and the reporting read replica exist specifically to stop one noisy tenant from becoming everyone else's incident.
Hosting architecture — Cloud edition
Every layer in the diagram above maps onto managed cloud services rather than hand-run infrastructure — the goal is nothing to patch at 3am that a managed service already handles.
LayerReference implementationWhy managed, not self-run
EdgeRoute 53 · ACM wildcard cert · ALB / CloudFrontDNS, TLS renewal, and edge routing are undifferentiated — a lapsed cert or a DNS misconfig is a self-inflicted outage.
App fleetECS Fargate (or EKS) · auto-scaling groupStateless containers scale on CPU/request metrics without capacity planning by hand.
Control planeElastiCache (Redis) · app-owned registry tableManaged Redis handles failover; the registry itself is just rows in the data tier below.
Data tierRDS Postgres (Multi-AZ) + read replica · S3Automated backups, patching, and failover — the single biggest source of 3am pages if self-managed.
Background servicesECS service (separate task def from web) · SQS-backed or Postgres-backed queueScales and deploys on its own schedule, independent of web traffic.
MonitoringCloudWatch / Datadog · PagerDuty on-call routingSame signals as Admin → System status, wired to actually page someone.
SecretsSecrets Manager / Parameter StoreDB passwords and API keys never sit in a repo or a plain .env in production.
Named after AWS here for concreteness; the same shape maps onto Cloud SQL/Memorystore/GCS on GCP or Azure Database/Cache/Blob Storage on Azure without changing the architecture.
Standalone / on-premise installation
The perpetual-license edition (see Pricing and Admin → License & billing) runs as a single-host Docker Compose stack instead of the multi-tenant cloud topology above — one customer, one install, no shared infrastructure to reason about.

License activation
A workspace's .trklic file — a JSON payload signed with Trakolo's private key — is issued whenever seats, modules, or the expiry date change, and verified locally against the public key bundled with the install. Air-gapped networks activate offline: License → Offline activation generates a request code, exchanged for a response code on the customer portal from any machine with outbound access, then pasted back in. See site/docs.html for the full walkthrough.
